MVC Architecture

  • client request a view
  • client invokes an action (change view, submit form)
  • server process the request in a controller
  • the request is satisfied
  • server sends a new view to the client

Server role

  • session affinity (via cookies)
  • authentication&authorization
  • validation
  • business logic
  • HTML rendering (JSP, freemarker, velocity, whatever)
  • date rendering
  • number rendering
  • batch jobs

Client role

  • render HTML page
  • send relevant cookies
  • optionally validation

Deployment model

  • one .war per webapp
  • one .war per servlet container
  • one JVM per servlet container

Perfectly reasonable architectures but...

What happens when...

  • a CSS attribute must be changed (e.g. because on IE the view is broken)
  • fix on a JS feature unavailable on iOS
  • internal server error
  • upgrade container (security fix)
  • JVM error (out of memory)
  • server must be rebooted (kernel update)

Problems

  • logger users must relogin after each deploy
  • stop adoption of continuous delivery
  • frontends and backend developers working on same artifact

Option #1: split webapps

separation of concerns

backend vs frontend

business logic vs presentation logic

Backend

backend as web service

  • session affinity (via cookies)
  • authentication&authorization
  • validation
  • business logic
  • HTML rendering (JSP, freemarker, velocity, whatever)
  • date rendering
  • number rendering
  • batch jobs

Backend development

can be developed using maven

optimal setup for backend developers

Frontend development

can be a single page application

can be developed using maven + webjars

but also with using node, grunt, gulp, bower

fully embrace the node.js ecosystem

Advantages

  • simpler code: less or none presentation logic in the backend
  • don't mess with merges
  • independent deploy
  • server side failures can be handled better (no 5xx pages or broken links)

Frontend as a static directory

can be deployed on any http server without service interruption (e.g. linux atomic ops on symlinks)

can be served as gzip (Content-Encoding:gzip)

can be served using sendfilei syscall (direct streaming file->socket)

One backend, Multiple frontend

simplifies development: frontend developer can use remote backend

customer can see new UI stuff quickly

same-origin policy

publish webservice to api.example.com, use it in demo1.myapp.com enabled by default in all major browser (IE9+) see http://enable-cors.org
