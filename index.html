MVC Webapp Architecture

  • client request a view
  • client invokes an action (change view, submit form)
  • server process the request
  • server sends a new view to the client

Server role

  • authentication&authorization
  • session affinity (via cookies)
  • validation
  • business logic
  • HTML generation (including dates, timezones, numbers, etc)
  • batch jobs
HTML generation using JSP, velocity, freemarker, JSF, whatever..

Client role

  • render HTML page
  • send relevant cookies
  • optionally validate data before send to the server

Deployment model

  • one .war per webapp
  • one .war per servlet container
  • one JVM per servlet container
  • reverse proxy in front of the servlet container
reverse proxy to terminate HTTPS connections, load balancing, etc

Perfectly reasonable architectures but...

What happens when...

  • a CSS attribute must be changed
  • fix on a JS feature unavailable on iOS
  • internal server error
  • upgrade container (security fix)
  • JVM error (out of memory)
  • server must be rebooted (kernel update)

Problems

  • logger users must relogin after each deploy
  • frontends and backend developers working on same artifact
  • stop adoption of continuous delivery

Simplify deploy

separation of concerns

backend vs frontend

Backend

backend as web service

  • session affinity (via cookies)
  • authentication&authorization
  • validation
  • business logic
  • HTML generation
  • batch jobs
server still must: - verify authentication - enforce security - validate input data (pre-conditions) e.g. @Valid - validate output data (post-conditions) e.g. unique data (404 vs 409)

Backend development

can be a mvc application view JSON as only view

can be developed using maven, gradle, lein, etc

optimal setup for backend developers

Frontend development

can be a single page application

can be developed using maven + webjars

but also with using node, grunt, gulp, bower

fully embrace the node.js ecosystem

Advantages

  • independent deploy
  • simpler code: minimal presentation logic in the backend
  • server side failures can be handled better (no 5xx pages or broken links)
  • more threads available for real work, not just serving static files

Frontend as a static directory

can be deployed on any http server without service interruption (e.g. linux atomic ops on symlinks)

can be served as gzip (for optimal bandwidth usage)

can be served using sendfile(2) syscall (direct streaming file->socket)

One backend, multiple frontends

customer can see new UI stuff quickly

can simplify development: frontend developers can use one remote backend

same-origin policy

publish webservice to api.example.com

use it in staging.exmachina.ch

use it in demo.exmachina.ch

see enable-cors.org

Simplify Caching

There are only two hard things in Computer Science:
cache invalidation
naming things
off-by-one errors
Be explicit about caching
(IE9 caches everything, even AJAX responses)
defaults to no caching
(using servlet filter or spring mvc interceptor)
Cache-Control: no-cache
Pragma: no-cache
then add cache headers only when proven useful
cache invalidation can be hard
